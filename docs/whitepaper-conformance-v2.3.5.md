# ASH Conformance Whitepaper — v2.3.5

## 1. What ASH Conformance Guarantees

ASH v2.3.5 ships 6 SDK implementations (Rust, Node.js, Python, Go, PHP, WASM) validated against 134 conformance vectors. Every vector has a fixed input and a hardcoded expected output generated from the Rust reference implementation.

**The guarantee:** Given identical inputs, all 6 SDKs produce byte-identical outputs for every conformance-tested operation — JSON canonicalization, query normalization, binding normalization, body hashing, client secret derivation, proof generation, scoped field extraction, unified proofs, and error behavior.

This means:

- A proof generated by the Python SDK can be verified by the Go SDK.
- A canonical JSON output from Node.js matches the Rust output character-for-character.
- An error triggered in PHP returns the same error code and HTTP status as the same error in WASM.

**What conformance does NOT guarantee:**

- **Security proof.** Conformance validates behavioral consistency, not cryptographic strength. It does not prove that ASH is resistant to all attack classes. A formal security audit evaluates threat models, timing properties, and protocol design — conformance vectors cannot.
- **Timing safety.** The suite tests that `timing_safe_compare("a", "a")` returns `true` and `timing_safe_compare("a", "b")` returns `false`. It cannot measure whether the comparison runs in constant time. Timing properties must be validated through dedicated side-channel analysis.
- **Network protocol correctness.** Conformance vectors test SDK functions in isolation. They do not test middleware integration, HTTP header parsing, or transport-layer behavior.
- **Completeness.** 134 vectors cover the known divergence points and critical edge cases. They do not exhaustively test every possible input. Fuzz testing and property-based testing are complementary tools for deeper coverage.

## 2. Design Decisions and Their Rationale

### 2.1 Path Percent-Decoding: `decode_all`

**Decision:** During path normalization, ASH decodes ALL percent-encoded characters, including reserved characters like `%2F` (→ `/`), `%3F` (→ `?`), and `%23` (→ `#`). The decoded characters then participate in dot-segment resolution and slash collapse.

**Risk:** This creates a path confusion surface. A proxy that routes based on the raw URL `/api/%2Fadmin` sees a single path segment, but ASH's canonical form resolves it to `/api/admin` — two path segments. If a security check happens on the raw URL and proof verification happens on the canonical form, they see different paths.

**Why it is locked in v2.3.5:** This is the Rust core's intentional behavior. All 6 SDKs are aligned to it. Changing to `decode_unreserved_only` would alter canonical output for any path containing encoded reserved characters, breaking existing proofs in the field.

**Mitigation for adopters:**

- Do not rely on percent-encoding to hide path segments from ASH. Ensure that routing and proof verification see the same resolved path.
- If your deployment sits behind a reverse proxy that normalizes paths differently, test your specific path patterns against ASH's canonical output.

**Future:** `decode_unreserved_only` is tracked as a candidate for v3.0, which would be a breaking behavioral change requiring regenerated vectors and a major version bump.

### 2.2 No Domain Prefixes in HMAC Inputs

**Decision:** In v2.x, HMAC message inputs use pipe (`|`) delimiters but do not include explicit domain prefixes. For example, the proof HMAC input is `{timestamp}|{binding}|{body_hash}`, not `ASH|proof|{timestamp}|{binding}|{body_hash}`.

**Risk:** Without domain prefixes, there is a theoretical cross-domain collision risk. If two different ASH operations happened to produce the same delimited string from different input types, they would generate the same HMAC. In practice, the input structures for client_secret, proof, scoped_proof, and unified_proof are structurally distinct (different field counts and types), making accidental collision unlikely.

**Why it is deferred to v3:** Adding domain prefixes changes every HMAC output for every operation. This is the most breaking change possible — it invalidates all existing proofs, all stored client secrets, and all test vectors. It cannot be done within a minor version.

**Mitigation for adopters:**

- Each proof domain (client_secret, proof, scoped, unified) uses a different HMAC key or a different number of delimited fields. Cross-domain collisions require an attacker to find inputs that produce identical delimited strings across structurally different formats.
- The `chain_hash` domain uses SHA-256 (not HMAC), providing additional structural separation.

**Future:** Explicit domain prefixes (e.g., `ASH-v3|proof|`) are planned for v3.0 as a first-class protocol change.

### 2.3 Chain Hash on ASCII Hex

**Decision:** The `chain_hash` is computed by SHA-256 hashing the ASCII bytes of the previous proof's lowercase hex string — not the decoded 32-byte binary value.

For example, if the previous proof is `a1b2c3...` (64 hex characters), `chain_hash = SHA-256("a1b2c3...")` where the input is the 64-byte ASCII string, not the 32-byte binary it represents.

**Why:** This eliminates an entire class of cross-language divergence. Converting a hex string to binary bytes requires each SDK to implement hex decoding identically — handling uppercase vs lowercase, odd-length strings, invalid characters, and memory allocation. By hashing the ASCII hex directly:

- The input is a plain UTF-8 string, which every language handles natively.
- There is no hex decode step that could diverge.
- The proof hex string is already the canonical representation (lowercase, 64 characters), so hashing it directly is hashing the canonical form.

**Trade-off:** The chain hash is 64 bytes of input instead of 32, making it ~2x slower for the hash computation. At SHA-256 speeds, this difference is negligible (nanoseconds).

## 3. Conformance Architecture

### Reference Implementation

The Rust core (`ashcore`) is the reference. All expected outputs in `vectors.json` are generated by running Rust functions and capturing their output. When any SDK disagrees with Rust, the SDK is fixed — not the vectors.

Rust was chosen for this role because of its deterministic memory model, strict type system, and absence of implicit runtime coercions (no silent float-to-int conversion, no locale-dependent string operations, no automatic type widening).

### Vector Structure

Each vector is self-contained:

```json
{
  "id": "json-001",
  "description": "Basic key sorting",
  "input_json_text": "{\"z\":1,\"a\":2}",
  "expected": "{\"a\":2,\"z\":1}"
}
```

Inputs are raw strings (not parsed objects) to ensure parser independence. Expected outputs are hardcoded — no formulas, no placeholders. Any SDK can validate by comparing its output string to the expected string.

### Error Vectors

Error vectors specify the exact error `code` and `http_status` that must be returned:

```json
{
  "id": "error-005",
  "description": "Empty binding rejected",
  "expected_error": { "code": "ASH_VALIDATION_ERROR", "http_status": 485 }
}
```

The error `message` content is non-normative — SDKs may use language-appropriate wording. Conformance tests validate code and status, not message text.

### Determinism Rules

The vectors encode these determinism constraints:

| Rule | Value |
|------|-------|
| Encoding | UTF-8, no BOM, no locale dependency |
| Unicode normalization | NFC before canonicalization |
| JSON key sorting | RFC 8785 (JCS), codepoint lexicographic |
| JSON duplicate keys | `last_wins` |
| Numeric precision | IEEE-754 double, RFC 8785 serialization |
| Plus sign in queries | Literal (never space), encoded as `%2B` |
| Hex output | Lowercase for hashes/proofs, uppercase for percent-encoding |
| Timestamps | Seconds (not milliseconds), digits only, no leading zeros |
| Path percent-decode | `decode_all` (including reserved characters) |
| Max JSON depth | 64 |
| Max payload size | 10 MB |

## 4. Scope and Limitations

ASH conformance is an engineering tool for cross-SDK consistency. It is not:

- A formal protocol specification (that would require a separate RFC-style document)
- A security certification (that requires an independent audit)
- A completeness guarantee (134 vectors cover known edge cases, not all possible inputs)

The conformance suite ensures that when you choose any ASH SDK, you get the same behavior as every other SDK. The security properties of that behavior — whether the HMAC construction is sound, whether the anti-replay mechanism is sufficient, whether the canonicalization is safe — are separate questions that require separate analysis.
