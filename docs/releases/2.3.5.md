# ASH Core v2.3.5 Release Notes

**Release Date:** 2026-02-07  
**Security Level:** Recommended for all users  
**Compatibility:** Backward compatible with v2.3.4

## Overview

This release includes security hardening improvements and naming convention compliance fixes based on comprehensive security audit findings. All users are recommended to upgrade.

## Naming Convention Compliance

### SDK Naming Convention Alignment

All function names now follow the [SDK Naming Convention](../../docs/development/SDK_NAMING_CONVENTION.md):

- **Public functions**: `snake_case` with `ash_` prefix (e.g., `ash_build_proof`)
- **Internal functions**: `snake_case` with `ash_` prefix (e.g., `ash_canonicalize_value`)
- **Deprecated aliases**: Documented as non-compliant, to be removed in v3.0

### Error Code Alignment

All error codes now match the [Error Code Specification](../../docs/reference/error-codes.md):

- Added `ASH_VALIDATION_ERROR` (HTTP 400) for general input validation
- `ASH_MALFORMED_REQUEST` kept for backward compatibility
- Complete alignment with specification v2.3.4

## Security Improvements

### SEC-AUDIT-005: HMAC Key Length Validation

**Issue:** HMAC-SHA256 accepts keys of any size, but keys larger than the block size (64 bytes) are hashed down internally. Extremely large keys could theoretically be used for DoS attacks.

**Fix:** Updated maximum nonce length:
- Maximum nonce length: 512 hex characters (256 bytes when decoded)
- Previous limit was 128 hex characters
- Error message: "nonce exceeds maximum length of 512 characters"

**Impact:** Nonces longer than 512 hex characters will now be rejected. This is well above the recommended minimum of 32 hex characters and should not affect any legitimate use cases.

### SEC-AUDIT-006: Sanitized JSON Parse Error Messages

**Issue:** JSON parse error messages from serde_json could contain fragments of the input payload, potentially leaking sensitive information in logs.

**Fix:** Error messages are now sanitized to a generic "Invalid JSON format" message that does not echo any user input.

**Impact:** Better security for production deployments where error messages may be logged or sent to monitoring systems.

### `#![forbid(unsafe_code)]` Guarantee

**Addition:** The crate now explicitly forbids unsafe code with compiler-level guarantees.

**Benefit:** Users can verify that 100% of the code is safe Rust with no unsafe blocks.

### New Error Code: ASH_VALIDATION_ERROR

**Addition:** Added `AshErrorCode::ValidationError` (ASH_VALIDATION_ERROR, HTTP 400)

**Purpose:** General input validation failures per ASH specification v2.3.4

**Breaking Change:** Replaced `AshErrorCode::MalformedRequest` (non-spec) with `AshErrorCode::ValidationError` (spec-compliant)

**Migration:**
```rust
// Before (non-compliant):
AshErrorCode::MalformedRequest

// After (spec-compliant):
AshErrorCode::ValidationError
```

## Migration Guide

### From 2.3.4 to 2.3.5

**Note:** This release includes one breaking change for error code compliance.

#### Breaking Change: Error Code Update

The non-compliant `AshErrorCode::MalformedRequest` has been removed. Use the spec-compliant `AshErrorCode::ValidationError` instead:

```rust
// Before (v2.3.4 and earlier):
AshErrorCode::MalformedRequest

// After (v2.3.5):
AshErrorCode::ValidationError
```

This aligns with the ASH Error Code Specification v2.3.4.

1. **Update dependency:**
   ```toml
   [dependencies]
   ashcore = "2.3.5"
   ```

2. **Verify no long nonces:** If you are using nonces longer than 512 hex characters (extremely unlikely), you will need to reduce their length.

3. **Error message handling:** If you were parsing JSON error messages, note that they are now generic:
   ```rust
   // Before
   "Invalid JSON: expected value at line 1 column 10"
   
   // After
   "Invalid JSON format"
   ```
